计算机网络-多媒体网络

2019-12-19

多媒体网络

多媒体网络应用

视频的性质

特点:

  • 高比特率(high bit rate).
  • 能压缩: 视频中存在两种冗余,可用视频压缩(video compression).
    空间冗余: 给定图像的内部冗余.
    时域冗余: 一副图像和后续图像的重复程度.

音频的性质

脉冲编码调制(Pulse Code Modulation,PCM)

  1. 模拟音频信号以固定速率采样.
  2. 采样值被量化(quantization)成一个有限个数值.
  3. 每个量化值由固定数量的比特表示.

压缩:

  • MPEG1 第3层(MP3)
  • 高级音频编码(Advanced Audio Coding, AAC).

注: 用户对于音频的失误更敏感.

多媒体网络应用的类型

分类:

  1. 流式存储音/视频.
  2. 会话式IP语音/视频.
  3. 流式实况音频/视频.

流式存储音频和视频

媒体是预先录制好的音,视频.

特点:

  1. 流(streaming): 用户从一个位置播放,同时从服务器接收后续的部分.
  2. 相互作用: 用户可对多媒体内容暂停,前进,倒退等操作.
  3. 连续播放: 根据初始记录的时序进行.

会话式IP语音和视频

因特网电话(Internet telephony)/IP语音(Voice-over-IP,VoIP): 因特网上的实时会话式语音.

特点:

  1. 时延敏感(delay-sensitive)
  2. 容忍丢包(loss-tolerant)

流式实况音频和视频

类似于广播的应用,有很多接收相同音视频节目的客户,可使用IP多播技术.

流式存储视频

分类:

  1. UDP流(UDP streaming)
  2. HTTP流(HTTP streaming)
  3. 适应性HTTP流(adaptive HTTP streaming)

客户缓存(client buffering): 用户请求视频到播放之间的视频储备.

UDP流

特点:

  1. 没有拥塞控制机制,以视频消耗速率传输.
  2. 使用实时传输协议(Real-Time Transport Protocol,RTP)的传输分组来传输音视频.
  3. 服务器和客户并行维护单独的控制连接,传输会话状态变化的命令.

缺陷:

  1. 恒定速率UDP不能提供连续的播放.
  2. 对每个进行的会话处理交互请求和跟踪状态,增加部署成本.
  3. 许多防火墙配置为阻塞UDP流量.

HTTP流

预取视频

预取(prefetching): 客户可以以高于消耗速率的速率下载视频,预取的视频存储在客户应用缓存中.

客户应用缓存和TCP缓存

服务器:
通过套接字进行发送,放置在TCP发送缓存中.

客户端:
应用程序从TCP接收缓存,并周期性的从缓存中抓取视频帧播放.

使用HTTP流时,一个满的客户应用缓存间接地对服务器到客户端能够发送的视频速率施加限制.

视频的早期中止和重定位

HTTP字节范围首部(HTTP byterange header): 指示客户当前要从所希望的视频中获取的字节范围.

重定向会导致客户缓存中的数据被抛弃,带宽和服务器资源被浪费.
应对:适用长度合适的客户应用缓存,或限制HTTP请求中使用字节范围首部预取的视频数量.

适应性流和DASH

经HTTP的动态适应性流(Dynamic Adaptive Streaming over HTTP, DASH): 视频编码成不同的版本,对应不同的比特率(质量).

特点:

  1. 允许客户使用不同的接入速率播放不同编码速率的视频.
  2. 端到端带宽改变时,DASH允许客户适应可用带宽.

实现:

  • 不同视频版本存储在HTTP服务器中.服务器有一个告示文件(manifest file),提供不同版本的URL和比特率.
  • 客户请求告示文件,从而选择合适的版本和字节范围.
  • 客户接收数据同时使用速度决定算法选择下一块.

内容分发网

必要性:

  1. 客户远离数据中心,其中一条链路的吞吐量小于视频的消耗速率,则带来时延.
  2. 流行视频可能在相同链路请求多次,浪费带宽.
  3. 单点故障导致无法完全无法使用.

服务器放置原则:

  • 深入: 在ISP中部署服务器集群来深入ISP接入网,减少端用户和集群之间的链路.
  • 邀请做客: 少量靠近第一层ISP的PoP位置建造大集群并使用专用高速网络连接.

CDN操作

截获和重定向:

  1. 用户访问Web网页,发送一个DNS请求.
  2. 用户的本地DNS服务器(LDNS)中继DNS请求.
  3. 目标DNS服务器返回一个CDN的主机名.
  4. 用户的LDNS向CDN主机发出DNS请求.
  5. CDN的DNS系统指向其内容服务器的IP地址.
  6. LDNS向主机转发内容服务器CDN结点的IP地址.
  7. 用户与该IP地址的服务器创建TCP连接发出HTTP GET请求.

集群选择策略

集群选择策略(cluster selection strategy): 基于客户的LDNS的IP地址将客户定向到CDN中服务器集群的机制.

策略:

  1. 地理上最为邻近(geographicaly closest): 将LDNS的IP地址映射成地理位置,选择地理位置最为接近的集群.
  2. 实时测量(real-time measuremetn): 对集群和用户之间的时延和丢包进行周期性测量.(发送探测分组;利用近期或正进行的流量特点；使用DNS请求流量测量时延)
  3. IP任播(IP anycast): 为每个集群指派向相同的IP地址,使用标准BGP从不同的汲取位置通告该IP地址.BGP路由器收到多个通告时,选择最好的路由.

IP语音

因特网电话(Internet telephony)/IP语音(Voice-over-IP): 经因特网的实时会话式语音.

尽力而为服务的限制

丢包

TCP消除丢包,时延不可接受.
VoIP默认运行在UDP中.

端到端时延

端到端时延(end-to-end delay):路由器中的传输,处理,排队时延等.

分组时延抖动

时延抖动(jitter): 由源到接收方之间经历的排队时延有波动.

消除: 使用序号(sequence number),时间戳(timestamp),播放时延(playout delay)来消除.

在接收方消除音频的时延抖动

策略:

  1. 固定播放时延: 在块产生的q ms后播放. 若块的时间戳为t,则接收者在t+q时播放.若约定的时间未到达,则被抛弃.
  2. 适应性播放时延: 估计网络时延及其变化,并在每个话音突峰期的开始相应地调整播放时延.

从丢包中恢复

丢包恢复方案(loss recovery scheme): 存在丢包的情况下试图保护可接受的音频质量.

方案:

  1. 前向纠错(Forward Error Correction,FEC).
  2. 交织(interleaving).

前向纠错

增加冗余信息来重建一些丢失分组的近似或者准确版本.

  1. 每发送n个块之后发送一个冗余编码的块,冗余块通过n个初始块的异或获得.当一个分组丢失,可重建该分组.多个分组丢失则不可重建.
  2. 发送一个较低分辨率的音频流作为冗余信息. 第n个分组包含第n个块加上第n-1个块的冗余信息.通过播放和后续分组一起到达的低分辨率编码块来隐藏丢失.

交织

发送方在传输之前对音频数据单元重新排序,使得最初相邻的单元在传输流中以一定距离分开,减少丢包的影响.(即一个报文中不再包含时间上相邻的数据,而是来自不同时刻的数据的整合)

差错掩盖

试图为丢失的分组产生一个与初始分组类似的替代物.

  1. 分组重复: 用丢失分组的前一个分组的副本替代丢失的分组.
  2. 内插法: 使用丢失之前和之后的音频内插形成一个合适的分组来隐藏丢失.

实时会话式应用的协议

RTP

RTP基础

在每个数据块的前面加上一个RTP首部(RTP header),数据块和RTP首部一起形成RTP分组(RTP packet).
在UDP报文段中封装该分组.将报文递交给IP.
接收端从UDP报文段提取RTP分组,从RTP分组中提取媒体块.
使用RTP分组首部字段来解码数据报.

特点:

  • 通常运行在UDP上.
  • 不确保数据及时交付.
  • 不提供服务质量保证(QoS).
  • 不保证分组的交付,或防止分组的失序交付.

RTP分组首部字段

组成:

  1. 有效载荷类型: 指示所使用的音视频编码类型.
  2. 序号字段: 用来检测丢包和恢复分组序列.
  3. 时间戳字段: 反映RTP数据分组中的第一个字节的采样时刻,去除分组时延抖动,提供接收方的同步播放.
  4. 同步源标识符(SSRC): 标识了RTP流的源.RTP会话中的每个流都有一个不同的SSRC.

SIP

会话发起协议(Session Initiation Protocol,SIP)

功能:

  1. 主叫和被叫经IP网络创建呼叫.
  2. 主叫确定被叫当前IP地址.
  3. 用于呼叫管理的机制.

特点:

  1. 可使用不同的套接字.
  2. SIP报文本身是可读的ASCII.
  3. SIP中所有的报文都要确认,可在UDP和TCP上运行.

示例

  1. 主要者发送INVITE报文,SIP会话开始.
  2. 被叫者收到INVITE报文,发送一个SIP响应报文.
  3. 主叫者发送SIP确认报文.
  4. 双方进行通信.

SIP地址

类似与电子邮箱地址或电话号码,名字等形式.
可以被包括在Web页面中.

SIP报文

  • Via首部: 设备的IP地址.
  • From首部行和To首部行.
  • Call-ID(呼叫标识符):表示该呼叫.
  • Content-Type(内容类型):定义SIP报文中的内容的格式.
  • Content-Length(内容长度): 提供报文中内容的字节长度.

名字翻译和用户定位

  1. 主叫者向SIP代理发送INVITE报文.
  2. 代理在SIP注册器进行DNS查询,并将报文转发给注册服务器.
  3. 注册器发送重定向应答.
  4. 主叫者的代理向被叫者的SIP注册器发送INVITE报文.
  5. 被叫者的注册器将INVITE报文转发给被叫者.
  6. 被叫者发送SIP响应给主叫者.
  7. 两者可以进行媒体发送.

支持多媒体的网络

支持多媒体应用的三种网络级方法

  1. 尽力而为服务.
  2. 区分服务.(不同类型的流量有不同类型的服务提供)
  3. 每连接服务质量(QoS)保证.(每个应用的实例显式地预约端到端带宽,具有确保的端到端性能)

定制尽力而为网络

多媒体应用的困难来源于其性能要求: 分组时延,时延抖动和丢包低.
解决: 提供充足的链路带宽.
宽带供给(bandwidth provisioning): 给定拓扑中为网络链路提供多大容量以取得给定水平的性能.
网络定制(network dimensioning): 如何设计一个网络拓扑中以取得给定水平的端到端性能.

提供多种类型的服务

将流量划分成多种类型,为不同类型的流量提供不同等级的服务.

几种见解

  1. 标记分组(packet marking):使得路由器区分属于不同类型流量的分组.
  2. 流量隔离(traffic isolation): 在流量类型之间提供流量隔离的度,以便一类流量不会受到另一类异常流量的负面影响.(1.流量监管:确保流量类型或流必须满足一定规则,设置监管机制确保准则被遵循;2.链路级的分组调度机制为每种类型发配固定量的链路带宽.)
  3. 当为流量类型或流之间提供隔离时,希望尽可能有效地使用资源.

调度机制

链路调度规则(link-scheduling discipline): 选择在链路上传输的排队分组的方式.

  1. 先进先出(FIFO): 调度规则按照分组到达输出链路队列的相同次序来选择分组在链路上传输.
  2. 优先级排队(priority queuing): 从队列为非空的最高优先级类中传输一个分组,同一优先级类的分组之间的选择以FIFO方式完成.(非抢占式优先级排队在高优先级分组到达时中断正在处理的低优先级分组)
  3. 循环和加权公平排队(round robin queuing discipline): 分组使用优先级排队一样的分类,但类之间不存在严格的服务优先级,循环调度器在这些类之间轮流提供服务.保持工作的循环规则(work-conserving round robin discipline):不允许链路空闲,在给定类中没有分组,会直接检查循环序列中的下一个类.
  4. 加权公平排队(Weighted Fair Queuing,WFQ): 每个类i被发配一个权,在类i有分组要发送的任何时间间隔中,第i类将确保接收到的服务部分为w_i/(sum(w_j))(分母是所有分组等待传输的类别之和)

监管:漏桶

3个监管准则

  1. 平均速度: 网络限制一个流的分组能够发送到网络中的长期平均速率.
  2. 峰值速率: 一个较短时间内能够发送的最大分组数.
  3. 突发长度: 限制极短的时间间隔内能够发送到网络中的最大分组数.

区分服务

因特网区分服务体系结构(Diffserv): 以可扩展性方式用不同的方法处理不同类型流量的能力.

组成:

  1. 边界功能: 分组分类和流量调节. 到达Diffserv服务器的分组的DS字段被设置为某个值,标识了该分组所属的流量类型,从而得到核心网络不同的服务.
  2. 核心功能: 转发.根据与分组类型相关的每跳行为(Per-Hop Behavior,PHB),该分组被转发到下一跳.

关键点:

  1. 标记和监管功能: 测定器依据一个或多个分组首部字段的值来选择分组,并做适当的标记.分组按照流量配置文件进行转发.若违反配置文件,则或被整形,或被抛弃.
  2. 每跳行为(PHB): 结点的转发行为描述.
    PHB使得不同服务类型获得不同性能.
    不强求使用特定机制.
    性能差异是可测量的.

PHB分类:

  1. 加速转发(Expedited Forwarding,EF): 流量离开路由器速率大于等于已配置的速率.
  2. 确保转发(Assured Forwarding,AF): 流量分类,提供确定的最低带宽和缓存.

每连接服务质量保证:资源预约和呼叫准入

如果充分的资源不总可用,并且要确保QoS,则需要一个呼叫准入过程,在这个过程中流要么被准入,要么被阻塞.

  1. 资源预留(resource reservation): 为呼叫发配所需的资源.
  2. 呼叫准入(call admission): 如果请求的资源不可用,则进行呼叫准入的呼叫被拒绝准入,否则完成呼叫准入.
  3. 呼叫建立信令: 信令协议协调本地资源的每跳分配,和呼叫在端到端路径熵的每台路由器中预约充分的资源.(呼叫建立协议(call setup protocol)).
